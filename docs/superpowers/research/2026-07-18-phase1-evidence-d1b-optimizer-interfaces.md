# d1b SUMMARY

SkillOpt (0.2.0, installed locally; adapter at benchmarks/src/pydocs_eval/optimize/optimizers/skillopt.py) mutates ONE flat markdown "skill document" string — never a dict: the trainer applies Edit ops {append, insert_after, replace, delete} with substring targets to skill_content, and the harness contract is an EnvAdapter subclass whose rollout(env_manager, skill_content: str, out_dir) returns list[dict] rows {"id": str, "hard": 0/1, "soft": 0.0-1.0}; the run is driven as a subprocess (run.py injects the adapter into scripts.train._ENV_REGISTRY, config YAML sections model/train/gradient/optimizer/evaluation/env) and emits <out_root>/best_skill.md, which the repo adapter parses back through the artifact's with_content()/validate() firewall. GEPA (not installed anywhere locally; interface web-fetched from gepa-ai/gepa main src/gepa/core/adapter.py) natively represents a candidate as Candidate = dict[str, str] (component_name -> text); its GEPAAdapter requires evaluate(batch: list[DataInst], candidate: dict[str,str], capture_traces: bool=False) -> EvaluationBatch{outputs, scores: list[float], trajectories} and make_reflective_dataset(candidate, eval_batch, components_to_update: list[str]) -> Mapping[component -> rows], with optional propose_new_texts(candidate, reflective_dataset, components_to_update) -> dict[str,str]. The repo's own prior art (ask-auto-optimization, PR #189) already solved exactly this dual-shape problem: every optimizer-facing artifact satisfies OptimizableArtifact {render() -> str, with_content(str), validate() -> tuple, fingerprint}, and the multi-section text artifacts (tool_docs, ask_prompt) store dict[section -> text] internally but serialize to ONE delimited document via render_delimited/parse_delimited with a CLOSED header set ("=== SERVER_INSTRUCTIONS ===", "=== SYSTEM_PROMPT ===", "=== REWRITE_PROMPT ===", "=== TOOL: <name> ==="), using header-collision detection instead of escaping; section naming precedent is SERVER_INSTRUCTIONS + "TOOL: {name}" per live tool, and scoring flows through FitnessFunction.evaluate(artifact, split) -> FitnessReport{score, components, cost_usd, n_samples}. Conclusion for Phase 1: dict[section_id -> text] maps bijectively onto GEPA's native candidate, and onto SkillOpt via the already-shipped render_delimited (SkillOpt sees the joined document; section headers double as stable edit anchors; parse-back + validate() firewalls boundary-mangling edits). Real impedance mismatches are enumerable and all have existing mitigations: SkillOpt can't do per-component selective updates (whole-doc only), its protected-region markers are hardcoded (can't freeze arbitrary sections in 0.2.0), the delimited grammar's closed _HEADER_RE must be deliberately widened for new section kinds (e.g., ROUTING or per-param sections — param descriptions currently have NO independent existence: server.py uses Field only for validation_alias/bounds, never description=), the render/parse newline normalization means fingerprints must hash the normalized surface, and a GEPA adapter needs per-sample scores whereas the repo's FitnessFunction returns one aggregate report (per-sample records exist internally in ask_rubric, so this is a plumbing change, not a redesign).

## Q4 — Does dict[section_id -> text] fit BOTH families?

**Yes, with the delimited-document renderer as the SkillOpt bridge — this is already the shipped pattern, not a new invention.**

- **GEPA**: bijective. `Candidate = dict[str, str]` IS dict[section_id -> text]; section ids become GEPA component names; `components_to_update` gives per-section selective optimization for free. A GEPA adapter must additionally supply per-example `scores: list[float]` per batch (EvaluationBatch), which the repo's aggregate `FitnessReport` does not expose — but ask_rubric already computes per-sample records internally (ask_rubric.py:131), so this is plumbing, not redesign. [GEPA half is web-derived.]
- **SkillOpt**: consumes ONE string. The bridge is exactly `render_delimited(dict) -> str` for the seed and `parse_delimited(best_skill.md) -> dict` + `find_header_collisions` + per-artifact `validate()` on the way back — precisely how `ToolDocsArtifact` serializes its dict state today (tool_docs.py:58-87) and how the skillopt adapter already firewalls output (skillopt.py:549-555). Section headers (`=== TOOL: grep ===`) are stable substrings, so SkillOpt's `insert_after`/`replace` edits can anchor on them.
- **Impedance mismatches (each with a concrete mitigation or cost):**
  1. **No per-component updates in SkillOpt** — it edits the whole document; there is no `components_to_update` analogue. Cost: SkillOpt runs optimize-everything; section freezing must be enforced post-hoc by `validate()` (reject candidates that changed frozen sections), not pre-hoc.
  2. **SkillOpt's protected regions are hardcoded** (`SLOW_UPDATE`/`APPENDIX` HTML-comment markers only, skill.py:27-30) — a Phase-1 source doc cannot mark arbitrary sections protected inside SkillOpt 0.2.0; the validate-firewall (mitigation 1) is the only lever.
  3. **Edits can mangle headers or smuggle new ones** — mitigated by design: a header-like line in content is promoted to a section and rejected as a collision (`_delimited.py:77-95`); a violating best yields `best=None` (skillopt.py:552-555). Never bypassed.
  4. **Closed header grammar must be deliberately widened** — `_HEADER_RE` (`_delimited.py:31-33`) enumerates every legal key; new Phase-1 section kinds (e.g., a `ROUTING` section or per-param keys like `PARAM: <tool>.<param>`) require extending this one regex + each artifact's `allowed` set. Small, single-source change, but it IS a grammar change and every existing delimited artifact shares the regex.
  5. **Param descriptions have no current storage** — nothing to externalize yet at parameter granularity (server.py grep above); introducing them means new sections AND a new render target (per-param `Field(description=...)` or continued in-prose), a Phase-1 design decision, not an optimizer constraint.
  6. **Newline/fingerprint normalization** — render→parse is not byte-stable on first pass (`_delimited.py:11-18`); the Phase-1 renderer must normalize (one parse/render pass) before hashing, matching the existing invariant.
  7. **Scoring-shape asymmetry** — SkillOpt wants per-item `{id, hard, soft}` dicts from a rollout it drives itself; GEPA wants `evaluate(batch, candidate)` per-example float scores under OUR driver; the repo Protocol returns one aggregate FitnessReport. One harness rollout layer can feed all three if it exposes per-sample (task_id, score) rows — the skillopt adapter's generated plugin already emits exactly that shape (skillopt.py:315-344).
