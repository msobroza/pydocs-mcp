## Fit assessment for pydocs-mcp Phase 1 (one markdown source → MCP descriptions + CLI help + system prompt)

What maps well:
- Frontmatter `id` per section = the section-ID requirement.
- Deterministic renderer + AUTOGENERATED sentinel + `sync --check` = the drift/round-trip guarantee Phase 1 wants.
- Format dispatch (`render_for(format)`) = one source, many outputs.

What does NOT map:
- No shipped format targets "MCP tool description", "CLI help string", or "server instructions" — all builtin formats emit harness config files (`claude_md_section`, `continue_rule`, etc.). A new adapter/format pair would need writing either inside the playbook repo or in pydocs-mcp.
- The engine's value-adds (repo detector, `applies_if` predicates, org overlay layers, eval runner) are irrelevant to rendering TOOL_DOCS/SERVER_INSTRUCTIONS.

Cost estimate (qualitative, unmeasured):
- Vendor/reuse whole engine: adds a 93-file alpha dependency (private repo, 0.1.0a0, no PyPI) plus jinja2+yaml (jinja2 already a pydocs-mcp required dep per CLAUDE.md deps list; pyyaml too) — dependency-wise cheap, but stability-wise poor (alpha API, private hosting breaks external contributors' installs) and functionally oversized.
- Borrow conventions only: pydocs-mcp already has the ingredients (jinja2, pyyaml, a `tool_docs.py` single source, function-local imports in server.py at lines ~183/249 per memory) — a small in-repo renderer (frontmatter-sectioned markdown → TOOL_DOCS dict + SERVER_INSTRUCTIONS + CLI help) reusing the playbook's frontmatter schema and sentinel convention is likely a few hundred lines plus tests, and keeps the frozen nine-tool contract logic local.

Recommendation: treat coding-agent-playbook as a design reference (accessible, readable, Apache-2.0 so copying code/conventions is license-clean), not as a dependency.
